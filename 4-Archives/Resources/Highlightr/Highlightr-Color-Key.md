# Highlightr Color Key – Single Source of Truth

## 1. Semantic Mapping (Option B – Canonical)

| Color  | Intended meaning                                      |
|--------|-------------------------------------------------------|
| Yellow | Key facts, premises, supporting evidence, main points |
| Green  | Definitions, explanations of terms, core concepts     |
| Blue   | Actionable items, instructions, next steps, thesis statements |
| Red    | Disagreements, critiques, warnings, questions, high-priority review |
| Orange | Examples, case studies, applications, analogies       |
| Purple | Quotes (direct verbatim), key insights, mental models/frameworks |
| Pink   | Ideas to resurface later, connections to other notes, open questions |
| Cyan   | Secondary/supporting details, less critical but useful |
| Grey   | Deprecated, outdated, or low-confidence info          |

## 2. Highlightr Plugin Usage & Storage Format (Technical Rules – MANDATORY)

**Current Highlightr settings (fill once):**
- Storage mode: **Inline CSS** ← (preferred)
- Or: CSS classes (only if you changed the default)

**Exact storage format the pipeline MUST use:**

**Inline CSS (recommended & default):**
- Yellow: `<mark data-highlight-source="agent" style="background: #FFF3A3A6;">text</mark>`
- Green:  `<mark data-highlight-source="agent" style="background: #C1E1C1A6;">text</mark>`
- Blue:   `<mark data-highlight-source="agent" style="background: #A3D8FFA6;">text</mark>`
- Red:    `<mark data-highlight-source="agent" style="background: #FFAAAAA6;">text</mark>`
- Orange: `<mark data-highlight-source="agent" style="background: #FFD9A3A6;">text</mark>`
- Purple: `<mark data-highlight-source="agent" style="background: #E6CCFFA6;">text</mark>`
- Pink:   `<mark data-highlight-source="agent" style="background: #FFC1CCA6;">text</mark>`
- Cyan:   `<mark data-highlight-source="agent" style="background: #A3FFFFA6;">text</mark>`
- Grey:   `<mark data-highlight-source="agent" style="background: #E0E0E0A6;">text</mark>`

**CSS Classes alternative (only if enabled):**
`<mark data-highlight-source="agent" class="hltr-yellow">text</mark>` (replace with your exact class names)

## 3. Agent vs user highlights (differentiation)

- **Agent-applied highlights** (distill-highlight-color, layer-promote, express-mini-outline): Every `<mark>` the pipeline writes **must** include `data-highlight-source="agent"`. That allows:
  - **CSS**: Target agent-only highlights, e.g. `mark[data-highlight-source="agent"] { border-left: 2px solid var(--text-muted); }` in a vault snippet to add a subtle left border.
  - **Search / Dataview**: Find agent-highlighted content by searching for `data-highlight-source="agent"` in source.
- **User highlights** (Highlightr plugin UI): The plugin inserts plain `<mark style="...">` or `<mark class="hltr-...">` with **no** `data-highlight-source` attribute. So any `<mark>` without that attribute is treated as user (or legacy).
- Highlightr still renders both the same by color; the attribute is optional for styling and filtering only.

**Gating / Enforcement Rules (the pipeline must obey these verbatim):**
- Native Obsidian `==text==` is **strictly forbidden** for any semantic highlight generated by distill.
- Never append `^{Color}` or any superscript/footnote syntax.
- Never mix formats in one note.
- Always reference this exact section when applying highlights.
- After applying, the note source must render correctly colored backgrounds in Reading mode via Highlightr.

**Legacy format:** `==text==^{Color}` is deprecated; pipeline outputs inline CSS only. Existing notes with legacy format may be migrated or left until next edit. The Visual Health Dashboard counts both formats for migration period.

**Vault default mapping** (updated): Yellow = key facts, Green = definitions, Blue = actionable, Red = warnings/critiques, Purple = quotes/key insights, Orange = examples, Pink = resurface/open questions, Cyan = secondary details, Grey = deprecated/low-confidence (matches table above).

**Example snippets** (optional but helpful for the LLM):
- Green example: "An **atomic note** is defined as a single idea expressed clearly." → `<mark data-highlight-source="agent" style="background: #C1E1C1A6;">atomic note</mark>`

## For MCP / pipelines

- **highlight_key** (in frontmatter or passed to skill): map span types to Highlightr color names, e.g. `{"key-terms": "Red", "quotes": "Blue", "actions": "Green"}`.
- **highlight_perspective** (in frontmatter or passed to skill): when set (e.g. `highlight_perspective: "combat systems"`), the skill applies **lens-focused** highlighting—spans that match the perspective get **analogous** colors (same theme = same hue family). Use for auto-shaping on distill/express runs. Example: note has `highlight_perspective: "performance"` → highlight performance-related phrases in blue–green analogous; contrast phrases in orange.
- Use **only** the format defined in **Section 2** above when writing back via `obsidian_update_note` / `obsidian_search_replace` (inline CSS or CSS classes from the key — never `==text==` or `^{Color}`).
- Apply only after **obsidian_create_backup** when doing in-place highlight edits.

## Project-Specific Guidelines

Skills (e.g. **distill-highlight-color**, **layer-promote**) use frontmatter **`highlight_key`** for project overrides; if absent, they fall back to the master key above.

**Examples by project**:

- **Project-X (user flows)**: Use blue–green analogous for user-flow ideas (continuity); orange contrast for pain points (tension).
- **Generic**: Use analogous colors for related ideas across linked project files; use complementary colors for contrasting or conflicting ideas so relations are clear at a glance.

**Color-callout border**: For **callout-tldr-wrap**, the TL;DR is wrapped in `> [!summary] TL;DR`. You can add a CSS snippet in the vault that styles this callout with a border color tied to a project `highlight_key` (e.g. project-themed left border) for visual consistency in hub views.
