# Question-led prompt crafter (Chat/Agent context rule)

- **Mode and first response**: **Use Chat or Agent mode** for this flow. Do **not** use Cursor Plan mode: in Plan mode the agent creates a `.plan.md` file instead of asking questions. Your **first** response must be to **ask** the first question from User-Questions-and-Options-Reference §1 (Q0 "Which kind?" if the user did not already say CODE or ROADMAP). Do **not** create or write a `.plan.md` file. Do not output a plan document before asking questions. **Even if the user is in Cursor Plan mode or you are instructed to create a plan**, for this flow your first response must be to ask the first question from §1. Do not create or write a `.plan.md` file. If the user wanted a plan document, they would not say "We are making a prompt" for this flow.
- **Trigger**: User says **"We are making a prompt"**, **"We are making a CODE prompt"**, **"We are making a ROADMAP prompt"**, or equivalent question-led crafting intent (e.g. "craft a queue entry", "build a prompt for ROADMAP").
- **Reference**: [Prompt-Crafter-Structure-Detailed](3-Resources/Second-Brain/Second-Brain-User-Flows/Prompt-Crafter-Structure-Detailed.md) (question-led entrypoints, Q&A pattern, protocol); [Prompt-Crafter-Param-Table](3-Resources/Second-Brain/Prompt-Crafter-Param-Table.md) (question_order, accepts_manual_text); [Prompt-Crafter-Implementation-Notes-V4](3-Resources/Second-Brain/Prompt-Crafter-Implementation-Notes-V4.md) (lock-first invariants, profile_derived, resume gate A–D, session safety); [Chat-Prompts](3-Resources/Second-Brain/Chat-Prompts.md) § Prompt crafting (question-led).
- **Known limitation (2026-03)**: `prompt_defaults.profiles` in Config currently contains only **roadmap** presets. CODE-side profiles (INGEST/ORGANIZE/etc.) are TBD and will be added after roadmap deployment testing. For CODE flows, when presenting the profile gate, state clearly in chat that profiles are tuned for roadmap and CODE runs rely on per-mode defaults until CODE-specific profiles exist; see Second-Brain-Config § prompt_defaults.
- **Questions source (mandatory)**: All questions and their **order** come **only** from [User-Questions-and-Options-Reference](3-Resources/Second-Brain/User-Questions-and-Options-Reference.md) **§1 (Question-led Prompt Crafter)**. The agent must **cover every param** in "Param order by branch" for the current branch in that order; question phrasing may be conversational (on-rails) but **param coverage and sequence** must match §1. **If the agent cannot read the file, or skips a param, or reorders params — abort and report failure** (see Fail conditions below).
- **Never skip a question.** For the current branch, you must ask every question §1 specifies (Q0 if needed, mode-narrowing, then every param in "Param order by branch," then manual text for each included text param, then Final). If you skip any of these, the run fails. Do not proceed to summary or queue append until all are asked and answered.
- **Optional = param inclusion, not skip.** Every param in §1 "Param order by branch" must be **asked** (one question per message); "optional" refers to whether the user includes the param (A/B/C), not to skipping the question.
- **Step 8 (queue append)**: Implement in line with the **Defenses** in the plan (validation before append, explicit routing, no write on decline, read-then-append, Task-Queue.md structure). See [Queue-Sources](3-Resources/Second-Brain/Queue-Sources.md) § Plan-mode queue routing and Task-Queue structure.
- **Question UI (Chat/Agent)**: In Chat or Agent mode, Cursor can show a **question box with click-to-answer (A, B, C)** when you ask **exactly one question per message** and use a format Cursor can parse. To trigger this UI and avoid display bugs: (1) **One question only** — in each message, output only one question and its options; do not include the next question or any other question; wait for the user to answer before sending the next message. (2) **Format:** Put the question text (and any "It does …" line) first, then put each option on **its own line** with a real line break: line 1 = "A. …", line 2 = "B. …", line 3 = "C. …". (3) **Use real newlines** — use actual line breaks in your reply, not the literal characters backslash-n. Do not wrap the question in a code block or other formatting that could cause newlines to render as literal `\n` (Cursor has a known bug that shows `\n` instead of newlines; using plain markdown with real line breaks reduces the risk). (4) Exactly three options (A., B., C.) per question; never D–G. Do not add commentary such as "Next I'll ask…" or "[Param 2 of 18]" in the same message; the message must contain only the one question and its A/B/C options (and optional brief "It does …" line).
- **Exactly three options only**: In this flow **every** question has **at most three options**, labeled **A.**, **B.**, and optionally **C.** Never use D, E, F, or G here. (Decision Wrappers and roadmap-next-step use A–G in other flows; this flow is A/B/C only.)
- **When user chooses C (AI reasoning)**: Limit your reasoning to **3–5 sentences max**. Append that exact text to the queue entry's **user_guidance** field (or **prompt** if no separate field) so pipelines receive it. See User-Questions-and-Options-Reference §1 "C choice behavior."

## Order invariant (Q&A before summary or queue)

- The **only** question that may be skipped is **Q0** ("Which kind?") when the user has already said CODE or ROADMAP. All other questions — mode-narrowing, every param in "Param order by branch," manual text for each included accepts_manual_text param, and the Final "Append to queue?" — must be asked.
- Do **not** create a plan document or write a `.plan.md` file and do **not** append to the queue until **all** of the following are complete: User-Questions-and-Options-Reference.md §1 read (step 1), schema loaded (step 2), kickoff resolved (step 3), mode narrowed (step 4), every optional asked and answered **in §1 order**, C choices resolved, and manual text phase completed for every included text-accepting param.
- Only after that may the agent: present summary, optionally present a short recap (questions + choices + payload), and append the payload to the queue.

## Fail conditions (prompt crafter)

- **Cannot read** `3-Resources/Second-Brain/User-Questions-and-Options-Reference.md` §1 → **FAIL.** Stop. Report: "Prompt crafter failed: could not read User-Questions-and-Options-Reference.md §1. Questions and order must come from that file." Do not ask questions from memory or from any other part of this rule.
- **Skips a question** that §1 specifies for the current branch → **FAIL.** Stop. Report: "Prompt crafter failed: skipped a required question. Follow the order in User-Questions-and-Options-Reference.md §1."
- **Skips a param** or **reorders params** or asks a question that **contradicts §1 semantics** → **FAIL.** Stop. Report: "Prompt crafter failed: param coverage or order violated. Follow User-Questions-and-Options-Reference.md §1 param order by branch and semantics."
- **Skips the Final question** (Message 9 / "Append to queue?" from §1) → **FAIL.** Always ask the Final question after presenting the summary; do not append without asking.
- **Creates or writes a `.plan.md` file** before all Q&A is complete → **FAIL.** Do not create or write any plan document for this flow.

## Behavior

1. **Load questions reference first (required)**: Read `3-Resources/Second-Brain/User-Questions-and-Options-Reference.md` and use **§1 (Question-led Prompt Crafter)** only. All questions and their order for this flow come from that section. **If you cannot read the file, FAIL** (abort, report "Prompt crafter failed: could not read User-Questions-and-Options-Reference.md §1", do not proceed).
2. **Load schema and scratchpad config**: Load base schema and defaults from [Queue-Sources](3-Resources/Second-Brain/Queue-Sources.md), [Parameters](3-Resources/Second-Brain/Parameters.md), [Second-Brain-Config](3-Resources/Second-Brain-Config.md) (including `prompt_defaults`), and read `crafting.tmp_prompt_path` / `crafting.max_reasoning_sentences` from Config. [Prompt-Crafter-Param-Table](3-Resources/Second-Brain/Prompt-Crafter-Param-Table.md) confirms question_order and accepts_manual_text per branch; the **exact question text and order** come from the reference file §1.
3. **Load or initialize tmp-prompt scratchpad**: At the **start of any crafting run**, attempt to read the JSON scratchpad at `crafting.tmp_prompt_path` (e.g. `.technical/tmp-prompt.json`).
   - **Expected keys**: `session_id`, `mode`, `asked`, `explicit_choices`, `agent_reasoning_log`, `final_payload_draft`; for V4 also `rail`, `rail_index`, and `profile_info` (with `active_profile` and `profile_derived` map: param → boolean; true = value from profile/default, false = manual/lock). If the file exists and parses into an object with at least the core keys, load it into memory for this run.
   - If the file does not exist or is empty, treat the scratchpad as fresh: `{ "session_id": null, "mode": null, "asked": [], "explicit_choices": {}, "profile_info": { "active_profile": null, "profile_derived": {} }, "agent_reasoning_log": [], "final_payload_draft": {} }` in memory; the implementation may choose to persist this structure once questions begin, but must never write user-authored PARA content.
   - If the file exists but JSON parsing fails or required keys are missing, **abort the crafter** for this run with a clear error (e.g. "tmp-prompt.json corrupt; please repair or delete and re-run") and **do not attempt to repair or overwrite** the file automatically. This keeps behavior predictable and avoids guessing from partial state.
   - **Mid-session restart (V4)**: If a valid unfinished session is detected (e.g. `rail_index` < rail length), offer the user: resume from next step, start fresh, or show summary then decide. If **start fresh**, create new in-memory state only; **do not overwrite** the scratchpad file until the new run successfully completes or is explicitly cancelled. Never blend old and new session state. See [Prompt-Crafter-Implementation-Notes-V4](3-Resources/Second-Brain/Prompt-Crafter-Implementation-Notes-V4.md) §4.
4. **Resolve kickoff**: If user did **not** specify CODE or ROADMAP, ask **exactly** the Q0 question and options from User-Questions-and-Options-Reference.md §1 ("Which kind?" with A. CODE … B. ROADMAP …). Do **not** guess. If you skip this question or use different wording, **FAIL**.
5. **Narrow mode (with RESUME-ROADMAP resume gate)**:
   - Ask **exactly** the mode-narrowing question(s) from §1 for the chosen branch (CODE or ROADMAP), in order, with **exact** wording and options from §1. Use only A, B, C. If you skip a question or ask a different one, **FAIL**.
   - When the branch is **ROADMAP → RESUME_ROADMAP**, before asking params 2–18, run the **resume gate** and **profile gate** per V4: if scratchpad has prior RESUME session with non-empty `explicit_choices`, show locked-params summary and offer A (keep locks and resume) or B (discard locks, start fresh). Then run **profile gate** (keep / switch / default / defer new profile). If user switches profile while locks exist, compute profile-vs-lock diff using `profile_derived` and offer options A–D per [Prompt-Crafter-Implementation-Notes-V4](3-Resources/Second-Brain/Prompt-Crafter-Implementation-Notes-V4.md) §3 (adopt and unlock affected, adopt but keep locked values, keep old profile, manually adjust locked params). **Lock precedence**: Values in `explicit_choices` **must always win** over profile or Config defaults; never overwrite a locked param with a profile default. See Implementation Notes §1 and §3.
   - Immediately after mode-narrowing (and, for RESUME_ROADMAP, after the gate(s)), continue with the **next** question in §1: the first param in "Param order by branch" that is not already fixed by mode or gate semantics. Do not stop after mode-narrowing.
6. **Ask optionals with scratchpad-aware skipping**:
   - For each param in the current branch’s "Param order by branch":
     - If the param name exists in the scratchpad’s `explicit_choices` map and the resume gate did not instruct a full re-evaluation that overrides sacred locks, **skip asking the A/B/C question entirely** for that param and treat the stored `explicit_choices[param]` value as the user’s current answer.
     - Otherwise, ask the A/B/C question as usual using the **exact** text and options from §1 (per-param and "It does …" as in §1).
   - Whenever a param is considered (asked or skipped), append its name to the scratchpad’s `asked` array for traceability.
   - For each question that is actually asked:
     - On **A-choice**, record the concrete value into `explicit_choices[param]` and mirror it into `final_payload_draft` for that param.
     - On **B-choice**, do **not** add the param to `explicit_choices`; update `final_payload_draft` so that the queue payload can either omit the key (default) or record that this param was considered and left at default.
     - On **C-choice**, do **not** add the param to `explicit_choices`; update `final_payload_draft` in the same way as B (omit or default), and also append a reasoning object to `agent_reasoning_log` of the form `{ "param": "<name>", "text": "<1–3 sentences>" }`, truncated to at most `crafting.max_reasoning_sentences` sentences.
   - For the current branch, you must still **cover every param** listed in §1; "skipping" here means "answer came from scratchpad sacred explicit_choices", not "question was never answered."
7. **Resolve C**: After **all** optionals are asked and answered, resolve remaining C semantics using defaults + context (current file, roadmap state), but **write the C reasoning text only into the scratchpad `agent_reasoning_log` and the eventual queue payload’s `agent_reasoning` field**, never into `user_guidance`. No additional user input is collected in this step.
8. **Manual text phase**: For **every** param that (a) was included via an A-choice and (b) has **accepts_manual_text** true, ask exactly one "What is the [param name]?" in §1 order using the **exact** wording from §1. Wait for confirmation before the next. Do not skip any of these; if you skip one, **FAIL**. For ROADMAP_MODE (setup), ask both project_id and source_file in order; do not skip because there are only two.
9. **Summary, plan (optional), and queue append**: Present the assembled payload for confirmation. **When assembling the payload:**
   - **Familial profiles (Config-Profiles):** **Always** run **`.cursor/skills/config-resolve-profile/SKILL.md`** during step 9 when assembling **`params`** for any mode that emits a **`params`** object (roadmap queue modes and other modes per Queue-Sources). Pass the **`params`** object after applying **`explicit_choices`** locks. **Do not** skip the resolver when familial keys are omitted: the skill applies the **default familial bundle** (`speed_mode: balance` + `repair_strategy: repair_first` + `validator_tier: forgiving`) so **effective** flat knobs and **`profile_derived`** match Layer 1 **A.2**. Those three labels are also documented as the **canonical default bundle** in **Second-Brain-Config** § **`profiles`** (Option B baseline); resolver behavior is unchanged. **Combo syntax** in string values (e.g. `speed_mode: "balance + repair_strategy: repair_first"`) is supported per the skill. Merge order: explicit queue keys > profile expansion > Second-Brain-Config > implicit defaults. Use **`profile_derived`** in the summary when values come from profile expansion (including default bundle) vs explicit locks.
   - **Lock precedence (V4)**: For every key in `explicit_choices`, the final `params` object **must** use that value. Never substitute from profile or Config for a key that is locked. See [Prompt-Crafter-Implementation-Notes-V4](3-Resources/Second-Brain/Prompt-Crafter-Implementation-Notes-V4.md) §1.
   - Apply the A/B/C → params mapping from User-Questions-and-Options-Reference §1: **A** = include explicit key = chosen value; **B** = omit key (use Config/MCP default); **C** = omit key + record reasoning in `agent_reasoning`.
   - **Final summary annotations**: For any param where the value is from `explicit_choices` and differs from the active profile default, label it in the summary (e.g. "param: value (manual override over profile default X)").
   - **For booleans:** Do not set explicit false on B or C — omit the key instead. Downstream treats absent = default (usually enabled).
   - Construct an `agent_reasoning` string (or array) from the scratchpad’s `agent_reasoning_log`, concatenating per-param reasoning snippets in a stable order; this field is intended for **AI reasoning only** and must never be mixed into `user_guidance`.
   - `user_guidance` remains **pure human input** (from the manual text phase or existing frontmatter); do not automatically inject C-choice reasoning into it.
   - Optionally output a plan that lists the questions asked and user choices. **Always** ask **exactly** the Final question from §1 (Message 9: A. yes — append | B. no — cancel | C. AI reasoning) after presenting the summary; do not append without asking. If you use different wording or skip the Final question, **FAIL**.
   - **If user declines (B / no / skip / copy only)**: Do not write to any file. State that the payload is in the plan for copy-paste or manual add.
   - **If user confirms (A / Y / yes / append)**:
     - **When mode is ROADMAP_MODE (setup)** (e.g. mode string "ROADMAP_MODE" for initial roadmap creation): (1) Validate and route the setup payload to `.technical/prompt-queue.jsonl`; read file, append the setup line, write back. (2) Commit scratchpad (e.g. set `rail_index` to rail length or mark session complete) so the next trigger does not treat this as mid-session. (3) Output **exactly** (V4 session-end): *"Queued successfully to prompt-queue.jsonl. Run **EAT-QUEUE** whenever you're ready to process it. Crafting session complete. If you need another one, just say 'We are making a prompt' again."* (4) **End the session** — do not ask any follow-up questions; do not continue into RESUME_ROADMAP. To craft a resume entry, the user starts a new crafting run and chooses RESUME_ROADMAP.
     - **When mode is RESUME_ROADMAP**: (1) Validate payload. (2) **Remove stale:** Read `.technical/prompt-queue.jsonl`; drop only lines that parse as JSON, have a `mode` field (string), and `parsed.mode === "RESUME_ROADMAP"`; keep all other lines (unparseable or different mode) in order. (3) Append the new RESUME_ROADMAP line to the filtered content, write back. If file is missing or unreadable, create minimal structure then write the new line. See Queue-Sources § Remove stale on RESUME_ROADMAP append. (4) Commit scratchpad; output the same V4 session-end message as above; **end the session** — no follow-up questions.
     - **For all other modes**: (1) **Validate** payload (mode present and known, params object if present, valid one-line JSON). If invalid, report error and do not append. (2) **Route** using the table: **Append to `3-Resources/Task-Queue.md`** only if mode is one of TASK_ROADMAP, TASK_COMPLETE, ADD_ROADMAP_ITEM, EXPAND_ROAD, REORDER_ROADMAP, DUPLICATE_ROADMAP, MERGE_ROADMAPS, EXPORT_ROADMAP, PROGRESS_REPORT; **otherwise append to `.technical/prompt-queue.jsonl`**. (3) **Read** current file contents, **append** exactly one newline and the JSONL line (for Task-Queue.md, append after body per Queue-Sources § Task-Queue structure), **write** back. If file is missing, create it (and parent for `.technical/`) with minimal structure. On write failure, report error and leave payload in plan; do not retry indefinitely. (4) Commit scratchpad; output the same V4 session-end message; **end the session** — no follow-up questions. No other vault writes.
   - **After any successful append (all modes):** Session ends. Use the same closing message (Queued successfully… Run EAT-QUEUE… Crafting session complete. If you need another one…). Do not ask "Which action?" or any further questions until the user triggers a new crafting run.

**Order**: Complete all of steps 1–7 (reference §1 read, schema loaded, kickoff, mode narrowed, all optionals in §1 order, C resolved, manual text done) before step 8. Do not create a plan document or `.plan.md` file or append to the queue until then.

### Payload shape and param_meta overlay (v2)

- **Primary payload (v1-compatible)**: The question-led crafter MUST continue to emit a primary queue payload that matches the existing contract in [Queue-Sources](3-Resources/Second-Brain/Queue-Sources.md): a JSON object with `mode` (string) and optional `params` (object). This primary payload is the **only** structure used for validation, routing, and behavior.
- **Optional `param_meta` overlay (v2)**: The crafter MAY, in addition to the primary payload, emit an advisory `param_meta` object:
  - `param_meta: { <paramName>: { description: string, defaultWhenC: string, usedBy?: string } }`
  - Param keys MUST match rows from [Prompt-Crafter-Param-Table](3-Resources/Second-Brain/Prompt-Crafter-Param-Table.md) / User-Questions-and-Options-Reference §1.
  - Values are populated from the **Param descriptors (for agents and param_meta overlay)** section in User-Questions-and-Options-Reference §1.
- **Advisory-only invariant**:
  - Queue processors and downstream pipelines MUST treat `param_meta` as **read-only decoration**. It is always safe to ignore; it MUST NOT change:
    - Whether a payload is considered valid.
    - How `mode` / `params` are interpreted.
    - Routing decisions (prompt-queue vs Task-Queue).
  - A valid v2 payload = a valid v1 payload (mode + params) **plus optional** `param_meta`. Consumers that do not know about `param_meta` remain correct by ignoring it.

## Invariants

- **No skips:** You must ask every question in §1 for the current branch, in the order given. Before step 8, mentally verify: kickoff (if needed), mode-narrowing, each param in "Param order by branch" (except any already answered by mode-narrowing), manual text for each included accepts_manual_text param, then Final. If any is missing, FAIL before appending.
- **Question wording and coverage**: Param **order and coverage** must match [User-Questions-and-Options-Reference](3-Resources/Second-Brain/User-Questions-and-Options-Reference.md) **§1** (kickoff, mode-narrowing, param order by branch, manual text phase, Final confirmation). Phrasing may be conversational (on-rails) as long as each param is surfaced in sequence and semantics (explicit / default / AI-choice) are preserved. For RESUME_ROADMAP profile-change flow, options A–D are allowed per Implementation Notes V4 §3. Ask one question (or one gate) per message. **Exactly three options** for standard param questions (A/B/C); RESUME profile gate uses A–D as specified in §1 and Implementation Notes.
- **Question output template**: For each question message: (1) First line: question text (e.g. "Which kind?" or "Include project_id?"). (2) Optional second line: "**It does:** …" when the param needs explanation. (3) Then a blank line (real newline). (4) Then "**A.** …" on one line, "**B.** …" on the next, "**C.** …" on the next. Nothing else in the message: no second question, no "Next I'll ask…", and no bracketed or preamble commentary (e.g. "[Param 2 of 18.]" or "Next: project_id") — only the one question and its A/B/C options (plus optional "It does …" line).
- **Fail on violation**: If the agent skips a param that §1 requires for the current branch, or reorders params, or asks a question that contradicts §1 semantics, the run **fails** (abort, report failure per Fail conditions, do not append to queue).
- **Single allowed write (plus scratchpad)**: The only **persistent queue write** in this flow is **appending the crafted payload** (one JSONL line) to the queue file (prompt-queue.jsonl or Task-Queue.md). No other vault writes to notes or PARA content are allowed. A **machine-only scratchpad** at the configured `crafting.tmp_prompt_path` (e.g. `.technical/tmp-prompt.json`) may be read and updated by the crafter during a session to track `session_id`, `mode`, `asked`, `explicit_choices`, `profile_info` (active_profile, profile_derived), `agent_reasoning_log`, and `final_payload_draft`; this file is treated as an implementation detail, excluded from Obsidian, and never contains user-authored content. When the user chooses **start fresh** at a mid-session restart, do **not** overwrite the scratchpad until the new run completes or is explicitly cancelled; never blend old and new session state.
- **No write on decline**: If the user declines to append (no / skip / copy only), do not write to any file; payload remains in plan for copy-paste.
- **Read-then-append only**: Append = read full file → append one newline + one JSONL line → write back. Never overwrite the file with only the new line; preserve all existing lines.
- Queue and crafting are **laptop-only**; mobile = observe + fill Ingest (see [Mobile-Migration-Spec](3-Resources/Second-Brain/Mobile-Migration-Spec.md)).

### Trigger shielding and partial-activation guard

- The trigger phrases for this rule (e.g. "We are making a prompt", "We are making a CODE prompt", "We are making a ROADMAP prompt") are **sensitive**: they enter a strict question-led mode with the Q&A contract above.
- If, at any point, the agent determines that it cannot continue to honor the full contract (for example, because the conversation has drifted into regular chat or required context is missing), it MUST:
  - Explicitly **exit prompt-crafter mode** (treat the rest of the exchange as normal chat),
  - **NOT** attempt to synthesize or partially apply a prompt-crafter queue payload from incomplete answers,
  - Optionally use the param descriptors section in User-Questions-and-Options-Reference §1 to explain params informally in chat (tooltips / explanations only).
- **Hard invariant**: Queue append, ROADMAP_MODE setup append, and RESUME_ROADMAP stale-removal are allowed **only** when:
  - The run has satisfied all Q&A requirements in this rule and in User-Questions-and-Options-Reference §1, **and**
  - The primary payload (`mode` + `params`) passes validation in Queue-Sources.
  - The presence or absence of `param_meta` can **never** by itself authorize writes or compensate for missing questions.
