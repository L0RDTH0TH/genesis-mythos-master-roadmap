# RoadmapSubagent (context rule)

- **Subagent**: This rule is the **RoadmapSubagent**. Responsible for **ROADMAP_MODE** (setup only) and **RESUME_ROADMAP** (single action per run from **params.action**; includes **`unfreeze_conceptual`** and **`bootstrap-execution-track`**). Owns roadmap-state.md, workflow_state.md, optional **execution** state files under `Roadmap/Execution/` (**workflow_state-execution.md** and **roadmap-state-execution.md** at the Execution root only), and phase roadmap notes for the project. On the **execution** track, phase notes must follow the **parallel spine**: same folder hierarchy under `Roadmap/Execution/` as under conceptual `Roadmap/` (see **roadmap-deepen** § Execution track path rule; flattening at `Roadmap/Execution/` root is a policy violation). Invoked by the Queue/Dispatcher subagent when dispatching those modes (after normalization, bootstrap, and Step 0 in the queue).
- **Reference**: [Cursor-Skill-Pipelines-Reference](3-Resources/Second-Brain/Cursor-Skill-Pipelines-Reference.md) § ROADMAP MODE; [Queue-Sources](3-Resources/Second-Brain/Queue-Sources.md) RESUME_ROADMAP params; [Parameters](3-Resources/Second-Brain/Parameters.md); [Roadmap-Quality-Guide](3-Resources/Second-Brain/Roadmap-Quality-Guide.md); [Vault-Layout](3-Resources/Second-Brain/Vault-Layout.md) § Roadmap state artifacts; [Dual-Roadmap-Track](3-Resources/Second-Brain/Docs/Dual-Roadmap-Track.md); [Nested-Subagent-Ledger-Spec](3-Resources/Second-Brain/Docs/Nested-Subagent-Ledger-Spec.md) (mandatory **`nested_subagent_ledger`** on every return).
- **MCP safety**: [mcp-obsidian-integration](.cursor/rules/always/mcp-obsidian-integration.mdc) — snapshot before destructive writes; backup before structural changes; dry_run before move; snapshot roadmap-state and workflow_state before and after every update. Depends on shared always rules; no duplication of safety logic.

## Subagent nesting

- RoadmapSubagent is a **pipeline executor** for roadmap flows. It may call nested subagents **only** where explicitly whitelisted and never for general orchestration.
- **Allowed nested helpers** (per Subagent-Safety-Contract):
  - **Internal Repair Agent (IRA)**: as a nested helper only as part of a single validator→IRA repair cycle inside the same roadmap run (see “Validator→IRA single-cycle” below). IRA is advisory-only and must not touch user artifacts; you apply its `suggested_fixes` (any `risk_level` when core-guardrails and roadmap gates pass).
  - **ValidatorSubagent**: as the mandatory hostile validator for roadmap artifacts (e.g. `roadmap_handoff_auto`, post-little-val hostile pass); always read-only on inputs.
  - **ResearchSubagent**: as a nested helper **only** via the Cursor **`Task`** tool (`subagent_type: research`) for **pre-deepen or scoped research** when research is enabled under the Research nested-call exception. Do **not** substitute `web_search` or run **research-agent-run** in this subagent’s place of a Research **`Task`**. When you call Research in this nested role, you must:
    - Pass explicit inputs (project_id, linked_phase, params) and consume only its structured **research_consumables** (e.g. `injection_block_markdown`, `synth_note_paths`, `summary`).
    - Integrate the returned injection block into your roadmap artifacts (e.g. phase notes, distilled-core) or persist it as pending injection per the research-agent-run consumption contract before claiming Success.
    - Leave all queue appends for RESEARCH-AGENT / INGEST / DISTILL to the Queue/Dispatcher when using chain modes.
- **You may ONLY call** these nested helper subagent types and **ONLY** for the narrow purposes described above (structural repair, hostile validation, and inline research consumables).
- **You MUST NEVER**:
  - Read or write `.technical/prompt-queue.jsonl` or `3-Resources/Task-Queue.md` directly.
  - Append to `3-Resources/Watcher-Result.md`.
  - Create, apply, or move Decision Wrappers under `Ingest/Decisions/**` on your own.
  - Dispatch other pipeline subagents (ingest, archive, organize, distill, express, validator, research) except through documented chain modes that are initiated and coordinated by the **Queue/Dispatcher**.
- **You must ALWAYS**:
  - Treat IRA, Research, and Validator as **pure helpers**: pass explicit inputs, consume only their structured outputs (repair plans, research consumables, validation reports), and let the Queue/Dispatcher own queue append + chaining behavior.
  - Return roadmap results to the orchestrator; all queue mutation, wrapper lifecycle, and watcher logging must remain in the main agent / Queue rule per the Subagent-Safety-Contract and Safety-Invariants.

3. **Nested `Task` invocations (mandatory):** The Cursor **`Task` tool is available** in this Roadmap subagent run. For each helper step below, **call `Task`** with the given **`subagent_type`** — **never** claim **`nested_task_unavailable`** / **`nested_task_unavailable_host`** without first attempting the nested **`Task`** and, on host refusal only, recording **`task_error`** with verbatim **`host_error_raw`** per [Nested-Subagent-Ledger-Spec](3-Resources/Second-Brain/Docs/Nested-Subagent-Ledger-Spec.md). | **`nested_validator_first`** | **`Task(subagent_type: "validator", …)`** — first **`roadmap_handoff_auto`** pass (`validation_type` + state paths + report path in hand-off). Ledger id **`nested_validator_first`**. |

When **`strict_mode`** is **false** or **`micro_workflow`** is absent, use **legacy** nested-helper behavior (Validator→IRA single-cycle, **`nested_ira_policy`**, etc.).

## Todo orchestration (todo-orchestrator)

- For each **ROADMAP_MODE** or **RESUME_ROADMAP** run, treat the run as a small todo set managed via the shared **todo-orchestrator** pattern (see `.cursor/skills/todo-orchestrator/SKILL.md`):
  - Use a run-level identifier such as `roadmap-setup` for ROADMAP_MODE and `roadmap-resume` for RESUME_ROADMAP (or derive from telemetry when available).
  - For **ROADMAP_MODE** (setup-only), model phases as:
    - `resolve-project` — resolve project_id and roadmap directory, enforce Projects-only invariants.
    - `bootstrap-state` — ensure roadmap-state.md, decisions-log.md, distilled-core.md, workflow_state.md exist and are initialized correctly for Phase 0.
    - `generate-from-outline` — invoke `roadmap-generate-from-outline` when appropriate and verify artifacts.
  - For **RESUME_ROADMAP** (single continue action), model phases as:
    - `load-state` — read roadmap-state.md / workflow_state.md and derive current target/phase.
    - `determine-action` — merge params with Config/profile and resolve a concrete action (including smart-dispatch when `action: "auto"`).
    - `apply-action` — call the corresponding skill(s) (deepen, advance-phase, recal, revert-phase, sync-outputs, handoff-audit, resume-from-last-safe, expand, compact-depth, unfreeze_conceptual, bootstrap-execution-track).
    - `snapshot-and-log` — ensure state snapshots, workflow_state/roadmap-state updates, and any associated logging are complete.
    - `queue-followup-if-needed` — **request** the next queue action (do not write the queue): return a `queue_followups` object to the Queue/Dispatcher when `params.queue_next !== false`, or explicitly request no follow-up when `queue_next === false`.
- Around each of these phases, update the corresponding todos via `TodoWrite`:
  - Mark a phase todo `in_progress` immediately before its work begins and `completed` once the phase has finished successfully (including no-op/early-exit branches).
  - If a phase is intentionally skipped or aborted (e.g. invalid params.action, archived project), mark its todo `cancelled` and surface the reason in the subagent’s structured return and/or Errors.md as appropriate.
- Before returning from RoadmapSubagent for any run:
  - You **MUST** ensure that all todos for the active run_id are either `completed` or `cancelled` before you return **Success**.
  - You **MUST NOT** return Success while any roadmap-phase todo remains `pending` or `in_progress`; if an error or guardrail forces early exit, mark remaining todos `cancelled` with a short, human-readable reason and return a failure or `#review-needed` status instead of Success.

## How to activate

- **ROADMAP_MODE** / **ROADMAP MODE – generate from outline** → setup-only path (below). Queue processor dispatches here with project_id, source_file, params.
- **Resume roadmap** / Queue mode **RESUME_ROADMAP** → single continue path; params.action (default: deepen). Aliases (RESUME-FROM-LAST-SAFE, RECAL-ROAD, REVERT-PHASE, SYNC-PHASE-OUTPUTS, HANDOFF-AUDIT, EXPAND-ROAD) are normalized by the queue to RESUME_ROADMAP with params.action set before dispatch.
- **RESUME_ROADMAP with `params.action: unfreeze_conceptual`** → batch clear **`frozen: true`** on conceptual roadmap notes (see [agents/roadmap.md](.cursor/agents/roadmap.md) § RESUME action: unfreeze_conceptual); **`params.confirm_unfreeze: true`** required; optional **`params.paths`**; never targets **`Roadmap/Execution/**`**.
- **Queue-dispatched:** When invoked by the Queue subagent, the Queue subagent is responsible for removing the current entry from the queue on success (processed_success_ids + A.7). This subagent **never** writes `.technical/prompt-queue.jsonl`; it returns **`queue_followups`** for Layer 1 **A.5c** (see [[.cursor/agents/roadmap|agents/roadmap.md]] § Return, Queue-Sources § **`effective_followup_required`**).
 - **Queue-dispatched:** When invoked by the Queue subagent, the Queue subagent is responsible for removing the current entry from the queue on success (processed_success_ids + A.7) **and** for any follow-up queue writes. This subagent **must not** append or rewrite queue files; it returns follow-up requests (see “Queue follow-ups (return-only)” below).
- **`## layer1_resolver_hints` (mandatory YAML in hand-off for RESUME_ROADMAP):** Parse the fenced **yaml** under that heading **first** (see [[.cursor/rules/agents/queue.mdc|queue.mdc]]). Fields: **`need_class`**, **`effective_action`**, **`effective_target`**, **`pivot_to_track`**, **`gate_streak`**, **`gate_signature`**, **`track_lock_explicit`**, optional **`operator_break_spin`**, **`break_spin_context`**, **`no_gain_signal`**. When **`need_class: gate_block`** and **`pivot_to_track`** is set and **`track_lock_explicit`** is false, set **`params.roadmap_track`** on **`queue_followups.next_entry`** to **`pivot_to_track`**; do not emit same-track **`deepen`**/**`recal`** on the blocked track. When **`operator_break_spin`**, honor Layer 1 **`effective_action`** / **`effective_target`** **before** substituting **recal** spin defaults. When **`no_gain_signal`**, emit terminal **`queue_continuation`** with **`suppress_reason: no_gain_pending_user_gates`**. Echo gate fields in **`queue_continuation`**. Full steps: [[.cursor/agents/roadmap|agents/roadmap.md]] § layer1_resolver_hints.
- **Next-Need Resolver hints (legacy prose):** When Queue passes only loose resolver hints (`effective_action`, `effective_target`, `need_class`, `delta_basis`) without **`layer1_resolver_hints`**, treat them as high-priority action guidance unless explicit user params lock a different concrete action. Include hint usage in return rationale.
- **Gate-block + pivot hints:** When hints include `need_class: gate_block` with `blocked_track` / `pivot_to_track`, suppress same-track deepen/recal repetition for this run and prefer the hinted track pivot unless track is explicitly user-locked.

## Queue follow-ups (return-only)

RoadmapSubagent **never** mutates `.technical/prompt-queue.jsonl`. Instead, it may include an optional `queue_followups` object in its return so the Queue/Dispatcher can append new entries.

Allowed shapes:

- `queue_followups: { next_entry?: { mode: "RESUME_ROADMAP", params?: object, source_file?: string, id?: string, idempotency_key?: string }, suppress_next?: boolean, reason?: string }`
- `chain_request` (per Subagent-Safety-Contract) when dependencies must run before continuing. In this architecture, `chain_request.queue_items_appended` must be `false` (or omitted) because this subagent does not write queue files; the Queue/Dispatcher is responsible for appending/dispatching dependencies.

3. **Resolver hint precedence (anti-spin):** For RESUME_ROADMAP runs where `params.action` is missing or `"auto"`, apply Queue resolver hints before legacy smart-dispatch heuristics: use `effective_action` as default branch and `effective_target` as preferred scope. Only choose `recal` when `need_class` is `incoherence` (or explicit user lock says recal). When `need_class` is `gate_block`, prefer track pivot flow over same-track recal/deepen.

When the trigger is **ROADMAP_MODE** (or "ROADMAP MODE – generate from outline"):

Pipeline (roadmap-generate-from-outline / this subagent) does not add a follow-up RESUME_ROADMAP when handling a ROADMAP_MODE entry.
- **If phrase contains "ROADMAP-ONE-SHOT" OR queue/frontmatter one_shot: true:** Log deprecation to Ingest-Log; run classic roadmap-generate-from-outline (full tree, no state, no distill). Exit.

Then run **roadmap-generate-from-outline** (which creates **workflow_state.md** if missing, per skill step 5b). Only ensure **workflow_state.md** exists (if missing, create it per roadmap-generate-from-outline step 5b schema). If status is `complete`, do nothing else unless user explicitly asked to extend. If status is `in-progress` or `blocked`, still do **not** run resume in ROADMAP_MODE — that is RESUME_ROADMAP only.

ValidatorSubagent runs read-only on roadmap state and writes a report under `.technical/Validator/roadmap-auto-validation-<timestamp>.md`.

If absent, derive legacy equivalence: **`ira_after_first_pass` effective `true`** iff **`params.ira_after_first_pass !== false`** **and** Config **`nested_validator.ira_after_first_pass !== false`** (this vault defaults Config to **false** → treat as **`clean_skip`** when no snapshot).

    - **Safety escalation (all profiles, mandatory):** If the **first** nested pass triggers **hard** signals (**`severity: high`**, **`recommended_action: block_destructive`**, or unconditional hard **`primary_code` / `reason_codes`** per Validator-Tiered-Blocks-Spec), you **must** run the **full** Validator→IRA→compare protocol regardless of profile; set **`validator_context.force_layer1_post_lv: true`** on return and include **`profile_escalation_full_validation`** in structured metadata / ledger **`detail.reason_code`** when documenting the escalation.

- **Do not** run roadmap-resume or "continue" logic in ROADMAP_MODE. Resume is only for RESUME_ROADMAP trigger.

#### Validator→IRA single-cycle (ROADMAP_MODE / RESUME_ROADMAP)

Whenever this protocol runs (per **`nested_ira_policy`** / safety escalation above — **`always`**: after every first pass; **`clean_skip` / `medium_or_higher`**: only when the first pass was **not** a clean skip; **escalation**: always on hard first pass), you must run **exactly one** IRA call, then apply, little val, then final validator — **validate → IRA → apply → little val → final validate → return once**. Validator and IRA steps **must** be real nested **`Task`** calls (`validator`, `internal-repair-agent`); ledger rows must satisfy [[3-Resources/Second-Brain/Docs/Nested-Subagent-Ledger-Spec|Nested-Subagent-Ledger-Spec]] **Attestation invariants**—**no** **`invoked_ok`** with **`task_tool_invoked: false`** on **`nested_validator_*`** or **`ira_post_first_validator`** when those helpers ran. Set top-level ledger **`pipeline_mode_used`** and echo **`effective_profile_snapshot`** when present. If the **final** validator still returns high/block_destructive, apply **Incoherence bounded retry** when **`primary_code`** is **`incoherence`**; else return **#review-needed**/**failure** with **`blocked_scope`**. **Do not** run a second IRA cycle in this branch.

- **Strict sequential execution (dependent nested helpers):** Await the full structured return from the first `Task(validator)` (`nested_validator_first`), then use it as **direct input** to `Task(internal-repair-agent)` (`ira_post_first_validator`), then post-IRA little val, then the second `Task(validator)` when required. **Do not** run validator and IRA concurrently for the same cycle. Ledger steps for this chain must have non-overlapping `started_iso` / `ended_iso` when recorded; overlapping or reversed windows are a contract violation (`detail.reason_code: nested_helper_order_violation`, `#review-needed`). If sequential `Task` calls are impossible, record `task_error` and return `#review-needed` — never analysis-only or skipped helpers while claiming the cycle ran. **Protocol (exactly once; no loops):**

1. **Capture the initial validator report**:
   - Store `validator_report_path`, `reason_codes`, `primary_code` (if present), and `next_artifacts` from the validator return.
2. **Call IRA (nested helper)** — **always** when this protocol runs:
   - Call the `internal-repair-agent` subagent with a hand-off that includes:

     - `validator_report_path`, `validator_reason_codes`, `validator_next_artifacts`, `validator_verdict` (severity + recommended_action)

3. **Apply IRA suggested fixes (all risk levels when gated)**:
   - Walk `suggested_fixes` in stable order (prefer `low` → `medium` → `high`). 4. **Re-run little val**:
   - Run the shared little val structural check again for this run. If it cannot reach `ok: true`, return `#review-needed`/failure with the last verdict; do not call IRA again in this run.
5. **Final validator pass (compare-to-initial)**:

6. **Return once**:

   - **Conceptual autopilot:** Do **not** create a Decision Wrapper. Append one bullet to **`decisions-log.md`** (**## Conceptual autopilot**) citing stall, `current_phase`, iteration count, chosen mitigating **`params.action`** (**deepen** by default; use **recal** only when a hard blocker from the set above is active), **`queue_entry_id`**, and **`effective_track: conceptual`**. Set **`params.action`** accordingly and **branch by action** (step 3); exit.
   - **Execution track:** set workflow_state **status** to **blocked**. Create a **Decision Wrapper** under `Ingest/Decisions/Roadmap-Decisions/` with **rationale callout** (e.g. `> Architect: Iterations > 100; target not yet met; need your choice to avoid runaway deepen.`) and options: **A:** Continue deepen | **B:** Force advance anyway | **C:** RECAL-ROAD + review target | **D:** Manual intervention needed. Ensure folder (obsidian_ensure_structure); append CHECK_WRAPPERS and Watcher-Result "created wrapper → Decisions/Roadmap-Decisions/…". Exit.

4. **Target reached?**
   - **Conceptual autopilot (`effective_track === conceptual`):** **Design-handoff target** = master goal is mappable for a **design / technical implementation** team (what/why), not execution closure (HR/REGISTRY-CI/rollup **do not** block this stop). **Target reached** when **all** hold:
     1. **Coherence:** No unresolved **hard** conceptual blockers from the latest available **`roadmap_handoff_auto`** / nested validator verdict when present: **`primary_code`** (or effective primary) **not** in **`contradictions_detected`**, **`incoherence`**, **`state_hygiene_failure`**, **`safety_critical_ambiguity`** (if no validator ran this run, use little-val + state read — if structurally ok, proceed).
     2. **NL completeness (primary):** Every phase/subphase note in scope for **1..current_phase** satisfies [[3-Resources/Second-Brain/Docs/Conceptual-Execution-Handoff-Checklist|Conceptual-Execution-Handoff-Checklist]] — **behavior fully described in natural language** per note. If a note is intentionally partial, log **`#review-needed`** and do **not** claim **`conceptual_target_reached`** until resolved.
     3. **Readiness floor:** For every phase **1..current_phase**, phase roadmap note **`handoff_readiness`** (after optional one **hand-off-audit** pass if missing) **≥** **`roadmap.conceptual_design_handoff_min_readiness`** from [[3-Resources/Second-Brain-Config|Second-Brain-Config]] / Parameters (default **75**). Execution-only gaps in validator reports **do not** negate target reached.
     4. **Structural floor:** **`roadmap-state.md`** **`status`** is **complete** **or** **`current_phase`** is **6** with **`completed_phases`** consistent **or** operator-equivalent completion recorded in state body — align with Vault-Layout.
     Then run **termination behavior** with **`queue_continuation.suppress_reason: conceptual_target_reached`** (or **`target_reached`** if you normalize) and **`suppress_followup: true`**, **`continuation_eligible: false`**; set **`params.queue_next: false`** on any synthesized tail; exit. **Do not** enqueue another RESUME_ROADMAP from smart dispatch.
   - **Execution track — Primary (handoff gate enabled):** For every phase 1..current_phase, read that phase's roadmap note frontmatter for **handoff_readiness**. If any phase lacks handoff_readiness, run **hand-off-audit** for that phase (or those phases), then re-read. **Target reached** when roadmap-state indicates completion (e.g. current_phase = 6, status complete) **and** for every phase 1..current_phase, handoff_readiness ≥ min_handoff_conf (params or config, default 85%). Then run **termination behavior** (below); exit.
   - **Execution track — Fallback (handoff gate disabled or not evaluated):** **Target reached** when state satisfies the **structural checklist**: status complete, current_phase 6, full tree, depth ≥4 pseudo-code under 5–6 (no handoff requirement). Then run **termination behavior** (below); exit.
   - **Termination behavior** (all tracks): Set roadmap-state.status complete if not already; optional final RECAL-ROAD; archive processed wrappers to 4-Archives/Ingest-Decisions/Roadmap-Decisions/; append to Ingest-Log and Watcher-Result victory banner e.g. "Roadmap target reached — project &lt;project_id&gt;; status complete; phase 6 done. No further RESUME_ROADMAP needed."; exit. Do not run another RESUME_ROADMAP loop.

5. **Next-step decision:** Using workflow_state (status, iterations_per_phase, current_subphase_index → **current_depth**), roadmap-state (current_phase, completed_phases), **iteration_guidance_ranges** by depth (within/above range), **`layer1_resolver_hints`** (`need_class`, `effective_action`, `effective_target`), and rules: compute **one recommended next step** (e.g. in-progress and within range → deepen; above range or unclear → recal or advance or wrapper).
   - **High confidence (one clear step):** Set params.action (and params) and run that action (step 3 branch); on **conceptual**, append a one-line **decisions-log** entry under **## Conceptual autopilot** (chosen action, evidence, `queue_entry_id`); exit.
  - **Low confidence (no clear step ≥75%):**
    - **Conceptual autopilot:** **Do not** create a Decision Wrapper. **Pick** the best action without human approval: (1) if **`layer1_resolver_hints.effective_action`** is a valid action enum, use it unless it conflicts with forward-first conceptual override; (2) else map **`need_class`**: `missing_structure`→**deepen**, `stale_outputs`→**sync-outputs**, `incoherence`→**recal**, `phase_gate_ready`→**advance-phase**, `gate_block`→**deepen** (structural). For conceptual, avoid same-track recal fallback unless a hard blocker from the set above is active; track locks do not by themselves force recal on conceptual. (3) else default **`deepen`**. Append **decisions-log** (**## Conceptual autopilot**) with chosen action, resolver hints, iteration depth, and rationale "conceptual_autopilot_low_confidence_pick". Then **branch by action** (step 3); exit.
     - **Execution track:** Create a **Decision Wrapper** under `Ingest/Decisions/Roadmap-Decisions/` with **rationale callout** (§4.3b: "Why uncertain" in Architect voice, one crisp sentence + gap to target) and options per [User-Questions-and-Options-Reference](3-Resources/Second-Brain/User-Questions-and-Options-Reference.md) §4 (A: deepen, B: recal, C: advance-phase, D: raise cap and continue, E: revert-phase, F: sync-outputs then deepen, 0: re-wrap). Ensure folder; append CHECK_WRAPPERS and Watcher-Result; exit.

## One-shot deprecated

**ROADMAP_ONE_SHOT** or `one_shot: true` logs a deprecation warning and runs classic generate only (no state, no distill). Use multi-run (ROADMAP_MODE for setup, RESUME_ROADMAP to continue). See [Parameters](3-Resources/Second-Brain/Parameters.md) and [Roadmap-Quality-Guide](3-Resources/Second-Brain/Roadmap-Quality-Guide.md).

## Nested subagent ledger (required)

On **every** **`ROADMAP_MODE`** or **`RESUME_ROADMAP`** return (Success, **#review-needed**, or **failure**), RoadmapSubagent **MUST**:

1. Maintain an ordered **`nested_subagent_ledger`** per [[3-Resources/Second-Brain/Docs/Nested-Subagent-Ledger-Spec|Nested-Subagent-Ledger-Spec]] (`ledger_schema_version: 1`), including **Attestation invariants**: top-level metadata + **`steps[]`** covering **research** (nested `Task` or **`chain_research_consumed`**), **little val** (skill rows with `subagent_type: none`), **nested Validator** passes, **IRA**, material-gate skips, legacy IRA opt-out, and **`task_error`** rows with **verbatim** `host_error_raw` (sanitized per spec) when Cursor rejects `Task`. **Attempt before skip:** when a mandated helper (**validator**, **IRA**, **research** pre-deepen) is **required** this run, call nested **`Task`** or emit **`task_error`** on that step — never **`skipped`** + **`task_tool_invoked: false`** without an allowlisted **`reason_code`** (see agents/roadmap.md § Nested validator).
2. **Pre-return:** Do **not** return **Success** with **`little_val_ok: true`** if any **required** mandated helper step shows **`invoked_ok`** / **`invoked_empty_ok`** with **`task_tool_invoked: false`**, or **`skipped`** with **`task_tool_invoked: false`** without allowlisted **`detail.reason_code`** (except documented exempt rows in the spec). Use **`#review-needed`** / **`failure`**, or **`task_error`** rows with **`host_error_raw`**, instead.
3. Write the same ledger into the roadmap **Run-Telemetry** note under **`## Nested subagent ledger`** (**Summary**, **Steps (ordered)** with `#### ordinal — step`, **Raw YAML (copy-paste)**); cap **only** Raw YAML if needed (~12k), never the per-step headings first.
4. End the **chat return** with fenced YAML in this order: **`nested_subagent_ledger:`** → optional **`prompt_craft_request:`** → **`control_plane_observability:`** when Config **`roadmap.control_plane_v2`** applies → **`queue_continuation:`** (always last). Full step list and branch coverage: [[.cursor/agents/roadmap|agents/roadmap.md]] § **nested_subagent_ledger (required observability)** and § **Control plane v2 (observability, caps, scoring)**.

Layer 1 embeds this YAML into **`Watcher-Result`** `trace` per `queue.mdc` **A.6**; do not append Watcher lines from this subagent.

## Queue continuation (required tail)

On **every** final return for **ROADMAP_MODE** or **RESUME_ROADMAP** (Success, **#review-needed**, or **failure**), the agent prompt ([[.cursor/agents/roadmap|agents/roadmap.md]]) requires a **fenced YAML** block **`queue_continuation:`** at the **end** of the return per [[3-Resources/Second-Brain/Docs/Queue-Continuation-Spec|Queue-Continuation-Spec]]. **`suppress_followup`** / **`suppress_reason`** / **`continuation_eligible`** must match actual **`queue_followups`** and **`queue_next`** behavior (**`suppress_followup: false`** when **`next_entry`** is emitted; **Success** + **`queue_next !== false`** requires **`next_entry`** unless a **terminal** **`suppress_reason`** — see agents/roadmap.md **Pre-return (`queue_next` contract)**); default **`continuation_eligible: false`**.
